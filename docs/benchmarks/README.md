# Benchmark artefacts

The raw markdown reports live in [`/bench/`](../../bench/) alongside the
scripts that generated them; this directory's job is to give them
context, link to the design doc that motivated each, and explain how to
reproduce.

All numbers here were collected on an AWS ``g5.xlarge`` node:

- GPU: NVIDIA A10G (Ampere, ``sm_86``, 24 GB HBM)
- Driver: 580.105.08, CUDA 13.0
- JAX: 0.10.0
- dtype: fp32 unless noted
- Reproducer: each script accepts ``--quick`` to run a small shape grid
  and writes its report alongside itself.

## Report index

### Substrate (Phase 2.A)

- [`/bench/PERF_SEMIRING_MATMUL.md`](../../bench/PERF_SEMIRING_MATMUL.md)
  -- Pallas Triton vs JAX ``lax.fori_loop`` for ``semiring_matmul``
  across the five differentiable algebras.  Headline: **6-16×
  speedup** of Pallas over JAX on REAL / TROPICAL / EUCLIDEAN at
  512² shapes; LOG is the slow algebra (0.93-1.98×) because its
  pytree-state accumulator inflates the K-loop body.  Generated by
  [`/bench/perf_semiring_matmul.py`](../../bench/perf_semiring_matmul.py).
  Linked from [`design/streaming-kernel.md`](../design/streaming-kernel.md).

- [`/bench/MEM_STREAMING_KERNEL.md`](../../bench/MEM_STREAMING_KERNEL.md)
  -- Empirical verification that the streaming kernel does *not*
  materialise the ``(M, K, N)`` value tensor.  Reads
  ``jax.devices()[0].memory_stats()`` before / after each call and
  compares to the analytical naive cost (which we don't run -- XLA's
  compile time on the materialised fusion blows past 60 s).
  Headline: at ``(512, 256, 512)`` the streaming paths peak at
  ~2.5 MB vs the naive analytical 256 MB.  Generated by
  [`/bench/mem_streaming_kernel.py`](../../bench/mem_streaming_kernel.py).
  Linked from [`design/streaming-kernel.md`](../design/streaming-kernel.md).

- [`/bench/G0_ELL_REPORT.md`](../../bench/G0_ELL_REPORT.md) -- The G0
  gate per IMPLEMENTATION_PLAN §3.1.  Records the
  ``confirm_pallas_falls_back()`` probe result (currently ``yes``:
  the policy is in effect) and the JAX-path baseline for
  ``semiring_ell_matmul`` over representative mesh-adjacency shapes.
  Headline: JAX path at ~12 G-events/s on REAL ``(m=4096, k_max=32,
  ncol=32)``.  Generated by
  [`/bench/g0_ampere_ell.py`](../../bench/g0_ampere_ell.py).
  Linked from [`design/ell-on-triton.md`](../design/ell-on-triton.md).

### Convolution (Phase 2 -- semiring_conv)

- [`/bench/PERF_SEMIRING_CONV.md`](../../bench/PERF_SEMIRING_CONV.md)
  -- Our ``semiring_conv`` vs cuDNN ``IMPLICIT_PRECOMP_GEMM`` (two
  baselines: TF32 tensor-core default and ``precision='highest'``
  strict fp32, so we isolate the tensor-core gap from the implicit-
  vs-explicit-im2col gap).  Headline: **1.7-1.9× slower than cuDNN
  fp32** for REAL on Ampere, matching the literature prediction of
  ~1.3-1.7× for explicit-im2col CUDA-core kernels (Chen et al.
  arXiv:2110.03901).  Generated by
  [`/bench/perf_semiring_conv.py`](../../bench/perf_semiring_conv.py).
  Linked from [`design/convolution.md`](../design/convolution.md).

## How to read these reports

Each report has the same structure:

- **Host**: device, platform, JAX version.  Numbers from a different
  generation (Hopper, Blackwell) or a different JAX pin shouldn't
  be directly compared.
- **Steady-state wall-time**: median of timed calls after warm-up.
  First-call compile cost is reported separately to avoid double-
  counting; per ``bench/_util.py``, ``BenchSample.warm_s`` is the
  number to quote when comparing implementations.
- **Compile cost**: first-call wall-time including XLA / Pallas
  tracing and lowering.  Useful for "should we ship this" decisions;
  not for steady-state comparisons.
- **Interpretation**: the editorial section that explains what the
  numbers mean and what would change them.  Read this first.

## Reproducing

From the repo root, with the editable install active:

```bash
python bench/perf_semiring_matmul.py --quick     # ~30 s
python bench/perf_semiring_matmul.py             # ~90 s
python bench/mem_streaming_kernel.py --quick     # ~20 s
python bench/g0_ampere_ell.py --quick            # ~15 s
python bench/perf_semiring_conv.py --quick       # ~45 s
python bench/perf_semiring_conv.py               # ~5 min
```

The scripts are deterministic given the random seed; numbers across
re-runs vary by a few percent due to GPU clock and thermal state.
For numbers worth quoting in a paper, run the full grid (not
``--quick``), pin the GPU clock with ``nvidia-smi -lgc <clock>``,
and report the median across multiple runs.
