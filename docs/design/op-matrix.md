# The op matrix: catalogue, probes, and the "turn it green" goal

> **TL;DR.** ``docs/op_matrix.md`` is the auto-generated status
> matrix for every public nitrix op: which JAX transformations it
> survives (jit / grad / vmap / jit∘grad), how it performs against
> a natural reference, and which algorithmic invariants the
> current implementation declares.  Generated by
> ``tools/op_matrix.py`` from a hand-curated catalogue +
> live probes + perf data scraped from ``bench/PERF_*.md``.
> Every cell is **observed on the generating host**; rendered
> Markdown carries the host snapshot so readers can calibrate.
> The goal is to turn every cell green; failures become the
> backlog.

## Motivation

We've shipped ~60-80 public ops across 10 subpackages.  Without a
single status board, it's easy for a regression to land in (say)
``stats.lme.flame_two_level`` that breaks ``jit(grad(...))`` and
go unnoticed because no test specifically asserts that combo.
The op matrix is the missing-link assertion: for every public op,
do the standard transforms work, end-to-end?

Hand-maintaining such a board would rot fast.  The auto-generator
runs live probes against the actual ops, so the rendered output
is always current with the code.

## Schema

Each cell records:

- **jit**: does ``jax.jit(op)`` succeed and produce a finite result
  on a representative input?
- **grad**: does ``jax.grad(scalar_loss(op))`` succeed and produce
  a finite gradient?
- **vmap**: does ``jax.vmap(op)`` succeed?
- **jit(grad)**: the canonical double-transform — does
  ``jit(grad(...))`` succeed?  This is where many subtle issues
  surface (cuSolver in autograd, while_loop in vmap, etc.).
- **perf ratio**: wall-time ratio nitrix / reference, scraped
  from ``bench/PERF_AUDIT.md`` and matched against an
  op-supplied ``perf_ref`` string.
- **invariants**: hand-curated tags for the algorithmic invariants
  the op exploits (sparsity, separability, FaST-LMM rotation,
  closed-form, etc.).
- **notes**: free-text annotation for caveats / known limitations.

## Hardware / CUDA / backend caveat

**Status cells are observed on the generating host.**  Concretely
on the nitrix test runner:

- ``jnp.linalg.eigh`` and ``jnp.linalg.qr`` are broken on the
  Ampere + CUDA 13.0 + cuSolver ABI mismatch.  Ops that route
  through ``safe_eigh`` (the cuSolver-CPU fallback) still pass
  the probes, but the underlying numerical path differs.
- ``jnp.linalg.cholesky``, ``svd``, ``lstsq``, ``solve``,
  ``solve_triangular`` are all working on this stack.

A different runner (Hopper / older CUDA / CPU-only) would produce
a different matrix.  For now we ship the matrix as a single-host
snapshot with the snapshot's metadata at the top.

**Future**: multi-host CI matrices.  Spin up one job per supported
hardware target (Ampere / Hopper / Blackwell / CPU-only); merge
their op_matrix.json outputs into a single colour-coded
"supported / broken / untested" table per cell.  Not in scope for
first cut; tracked under ``open questions``.

## Adding a new op to the matrix

1. Add a public symbol with a docstring, jaxtyping signature, and
   tests as usual.
2. Add an ``OpInfo`` entry in ``tools/op_matrix.py``:
   - ``qualname``: ``nitrix.subpackage.symbol``.
   - ``fixture``: zero-arg callable returning ``(args, kwargs)``
     for a representative call.
   - ``diff_arg`` / ``vmap_arg``: indices into ``args``, or
     ``None`` if not applicable.
   - ``invariants``: free-text tags.
   - ``perf_ref`` (optional): a string that matches the op's
     name in the perf-audit output, so the wall-time ratio gets
     joined into the matrix.
3. Rerun ``python tools/op_matrix.py``; the regenerated
   ``docs/op_matrix.md`` should include the new row.

## What the probes cannot catch

- **Semantic correctness**: a cell can be all-green and still
  compute the wrong thing.  That's what the regular test suite
  is for; the op matrix only tests *that the transformations
  run*, not *that the results are correct*.
- **Performance at scale**: the probes use tiny fixtures so the
  matrix runs in seconds.  Perf characterisation comes from the
  separate bench scripts (``perf_audit.py`` etc.).
- **Memory ceilings**: not yet captured.  Would require a
  per-op HLO-shape audit; a follow-up.
- **Differentiability under degenerate inputs**: the probes use
  generic random inputs; a separate "stress test" probe would
  use ill-conditioned / near-degenerate inputs.

## Open questions

- **Multi-host CI**.  How to merge matrices from multiple runners
  into a single table without losing information about which
  cells are broken on which hardware.  Could render as side-by-
  side or stacked tabs; needs a UX decision.
- **Op-level perf budgets**.  The matrix records wall-time
  ratios but doesn't gate on them.  A future version could
  declare per-op budgets ("rbf_kernel must be < 5x sklearn") and
  fail CI if exceeded.
- **Invariant verification**.  Currently the "invariants" column
  is hand-curated free-text.  Some invariants could be machine-
  verified via HLO audit (e.g., "no V*N^2 intermediate") --
  bridging the lme / lobpcg tests' existing audits into the
  matrix framework.
